# 배치 프로젝트 개발 지침

이 파일의 규칙은 모든 코드 작성과 수정에 적용된다.
규칙과 다르게 구현할 이유가 있으면 진행하기 전에 사람에게 확인한다.

## Java

- `var`는 한 줄만 봐도 타입이 분명할 때만 쓴다.

## Test

- JUnit 5와 AssertJ로 작성한다.
- 가능하면 애플리케이션 컨텍스트를 띄우지 않고 객체를 직접 생성해 단위 테스트한다.
- 목(mock) 라이브러리 대신 인터페이스를 구현한 가짜 객체를 만들어 검증한다.
- 정상 경로만 확인하지 말고, 예외가 발생해야 하는 경우와 경계값도 함께 검증한다.
- 테스트 메서드의 의도는 `@DisplayName`으로 표기한다.

## 재시도(retry) 규칙

- 재시도는 `RetryTemplate`을 직접 호출해서 구현한다.
- 재시도 대상 동작은 멱등(idempotent)해야 한다. HTTP GET 메서드를 호출하는 경우는 멱등하다고 가정하고, 나머지는 사람에게 물어서 확인한다.
- 정책 값(간격·횟수·최대 시간)은 추측해서 채우지 말고, 비어 있으면 사람에게 먼저 묻는다.
- 모든 재시도가 실패하면 마지막 예외를 삼키지 말고 그대로 전파한다.
- 의도한 횟수만큼 실패한 뒤 성공하는 가짜 구현을 만들어 실제 호출 횟수를 검증하는 테스트를 함께 작성한다. 정상 경로만 통과하는 테스트로 끝내지 않는다.
